## 项目背景(供 AI 理解上下文)

我在做一个第一人称视角(AI 眼镜拍摄)的实时**四川麻将**识别系统,采用两阶段架构:
- 检测器:YOLO11(2 类:`tile_face` 牌面 / `tile_back` 牌背)
- 分类器:29 类牌面分类(见下方类别表)

**重要约束:本项目只针对四川麻将(血战到底),牌池只有万/条/筒共 108 张,不含东南西北中发白等字牌,也不含花牌。** 全项目类别体系按此设计。

Phase 0 的目标是建立**冻结测试集 + 自动化评测体系**,所有后续训练以此为唯一标尺。
**测试集图像全部来自抖音/B站上指定博主的四川麻将实拍视频**(手元/第一人称视角类博主),不做自拍。我负责提供博主清单、平台 Cookie 和标注工作;你(AI)负责抓取、筛选、评测的全部工具链代码。
**标注工具统一使用 X-AnyLabeling**(免费开源桌面应用,单机使用,无需部署服务;支持加载 ONNX 模型辅助标注、内置 SAM、原生导出 COCO)。所有"预标注"均生成 X-AnyLabeling 兼容格式(每图一个同名 JSON)。
预留说明:待我的 AI 眼镜可用后,会另建 `test_set_v2_glasses`(眼镜实拍域),届时所有评测脚本需支持对多个测试集分别出报告——本期实现时就按"测试集路径可配置"来设计,不要写死。

### 类别表(全项目统一,写入 `configs/classes.yaml`)

```yaml
# 检测类别(2 类)
detection:
  0: tile_face
  1: tile_back

# 分类类别(29 类,四川麻将)
classification:
  - [w1, w2, w3, w4, w5, w6, w7, w8, w9]   # 一万~九万
  - [t1, t2, t3, t4, t5, t6, t7, t8, t9]   # 一条~九条
  - [b1, b2, b3, b4, b5, b6, b7, b8, b9]   # 一筒~九筒
  - [back]      # 牌背
  - [unknown]   # 侧面/严重遮挡/不可读/非万条筒的牌
```

**字牌处理规则**:实体麻将套装里通常带字牌,网络视频里也可能偶尔出现。凡画面中出现东南西北中发白或花牌:检测仍标 `tile_face`(它确实是一张牌面),分类一律标 `unknown`。线上推理时 `unknown` 结果直接丢弃,不参与投票。

## 任务 1.5:博主视频批量抓取 `scripts/fetch_videos.py`

**做什么**:读取 `configs/sources.yaml`,把每个博主的视频批量下载到 `data/raw_videos/{平台}_{UID}/`。

**功能需求**:
1. **B站**:用 yt-dlp 直接抓 UP 主空间(支持传入空间 URL 或 UID),720p 优先(`-f` 选择合适清晰度,不需要 4K,省空间省带宽)
2. **抖音**:yt-dlp 对抖音支持不稳定,优先集成 `f2` 或 `TikTokDownloader`(JoeanAmier 开源项目)这类专用工具作为后端;需要我提供登录 Cookie 的,在 README 写清楚 Cookie 获取步骤(浏览器 F12 复制即可的程度),Cookie 放 `configs/cookies/`(gitignore)
3. **标题关键词过滤**:支持 `include_keywords` / `exclude_keywords`(在 sources.yaml 每个博主条目下配置),如排除标题含"教学""解说""线上"的视频,只抓实战类
4. **增量下载**:维护 `download_state.json`,记录每个博主已下载的视频 ID,重复运行只抓新视频
5. **限速与礼貌**:请求间隔随机 3~8 秒,单博主单次最多抓 `max_videos` 个(默认 20),失败重试 2 次后跳过并记录
6. **元数据留存**:每个视频旁存一个同名 `.info.json`(标题、发布日期、URL、博主),后续追溯用
7. 输出 `fetch_report.json`:每个博主抓了几个、跳过几个、失败原因

**验收标准**:对一个 B站 UP 主(我提供 UID)实际运行,能增量抓取并正确命名;抖音侧给出完整可运行的方案,我配好 Cookie 后单博主能跑通;中断后重跑不重复下载。

**人工配合**:我提供 6~8 个博主的 UID/主页链接填进 sources.yaml,提供抖音 Cookie;抓取过程无需人工值守。

**给 AI 的提示**:抖音的抓取工具生态变化快,实现前先查目标工具当前是否可用、API 是否变更;若 `f2` 不可用,降级方案是生成"待下载视频 URL 清单",我用第三方下载器手动批量下,脚本只负责后续整理归档——保证整体流程不被卡死。

## 任务 2:智能抽帧脚本 `scripts/extract_frames.py`

**做什么**:从 `data/raw_videos/` 批量抽帧,但不是简单定时抽,要做质量与多样性筛选。

**功能需求**:
1. 基础抽帧:默认每秒 0.5 帧(`--fps` 可调),用 OpenCV 或 ffmpeg
2. **糊帧过滤**:Laplacian 方差 < 阈值(`--blur-threshold`,默认 100)的帧丢弃,阈值要在文档里说明如何用样本帧校准
3. **曝光过滤**:平均亮度 <30 或 >225 的帧丢弃
4. 输出命名:`{视频文件名}_f{帧号:06d}.jpg`,保留原始分辨率
5. 输出一份 `extract_report.json`:每个视频抽了多少帧、丢弃多少及丢弃原因分布

**验收标准**:对一个 1 分钟测试视频运行,输出帧无明显糊帧;报告 JSON 字段完整;处理速度 ≥ 实时(1 分钟视频 ≤1 分钟处理)。

## 任务 2.5:网络视频筛选辅助工具 `scripts/screen_web_videos.py`

**做什么**:任务 1.5 抓回来的视频数量会远多于最终需要的。此脚本自动粗筛,把"有效片段"挑出来,大幅减少我人工看片时间。

**功能需求**:
1. **含牌检测**:对每个视频按 1 fps 采样,用我提供的旧 YOLO11 模型(`--model` 传入)跑推理,统计"检出 ≥3 张牌的帧"占比;占比 < 30%(可调)的视频整体标记为"低价值"
2. **录屏排除启发式**:检测画面是否为线上麻将录屏——特征包括:画面四周存在大面积纯色/渐变 UI 边框、帧间背景完全静止(背景区域像素方差≈0)、检出的"牌"长宽比异常统一且排列成完美网格。命中则标记 "疑似录屏"
3. **有效片段切割**:对保留的视频,找出连续"含牌帧"区段(允许 3 秒间断),用 ffmpeg 无损切出片段(`-c copy`),输出到 `web_clips/`,命名 `{原名}_clip{n}.mp4`
4. 输出 `screen_report.json` + 一个简单 HTML 预览页:每个视频/片段一行,附 3 张缩略图 + 判定结果 + 判定理由,我只需快速过一遍预览页做最终取舍
5. 字牌提示(尽力而为):若旧模型类别表里有字牌类且某视频字牌检出占比高,在报告中标注"疑似非四川麻将",供我人工确认

**验收标准**:对 1 个真实麻将视频 + 1 个无关视频 + 1 个线上麻将录屏(我会提供样例)运行,三者判定正确;切出的片段可正常播放。

**人工配合**:我负责搜索和 yt-dlp 下载,然后看预览页定取舍,单个视频决策 <10 秒。

## 任务 3:去重与多样性采样 `scripts/dedup_filter.py`

**做什么**:从 `frames_candidate/` 中选出最终待标注的 400~500 张,存入 `frames_selected/`。

**功能需求**:
1. **感知哈希去重**:imagehash(phash),汉明距离 ≤ 8 视为重复(阈值可调),同组只保留 Laplacian 方差最高的一张。注意:同一博主的视频机位/桌布固定,跨视频重复率高,phash 要跨整个博主目录比对,不只在单视频内
2. **博主配额均衡**:按博主(目录名 `平台_UID`)分配总量配额,默认均分(`--total 500` ÷ 博主数),单个博主占比上限 25%,避免测试集被某一张桌子主导;每张图的来源博主写入文件名前缀,并最终进入 COCO 的 image 自定义字段 `source`(值 = `平台_UID`)
3. **单视频上限与时间分散(关键)**:同一视频内桌布/光照/机位完全相同,帧间高度相关——单视频选中帧数上限默认 8(`--per-video-cap`),且入选帧时间间隔 ≥30 秒(`--min-gap-sec`),优先保留牌局状态差异大的帧(候选阶段按时间均匀分布即可近似);保证没有某个视频独占
4. 输出 `selection_report.json`:每个博主、每个视频的候选数/选中数,以及总体统计
5. dry-run 模式:只出报告不复制文件

**验收标准**:对包含重复帧的目录运行后,人眼抽查无近似重复;各场景组都有样本;报告与实际文件数一致。

## 任务 5:COCO 校验脚本 `scripts/validate_coco.py`

**做什么**:标注完成并经 export_to_coco.py 转换后,自动检查标注质量问题。

**检查项**:
1. 格式合法性:能被 pycocotools 加载;类别 ID 与 `classes.yaml` 完全一致
2. 异常框检测并输出清单(图名 + 框坐标 + 问题类型):
   - 面积 < 25 px² 的极小框(可能是误触)
   - 宽高比 > 5 或 < 0.2 的异常长条框
   - 完全相同的重复框(IoU > 0.95 且同类)
   - 超出图像边界的框
3. 统计报告:总框数、每类框数分布(检出某类为 0 要警告)、每图平均框数、按框尺寸分桶(<20px / 20~40px / >40px,以框的较短边计)的数量分布
4. **可视化抽查**:随机抽 30 张图把框和类别画上去,输出到 `validation_preview/`,供我人眼复核

**验收标准**:对一份故意做错的标注文件(你自己构造测试用例)能全部检出;预览图类别文字清晰可读。

## 任务 6:评测体系(本 Phase 核心交付)

### 6a. `eval/eval_detection.py` — 检测评测

输入:COCO 格式 GT + COCO 格式预测结果(detections json)
输出指标:
- 标准 COCO mAP@0.5、mAP@0.5:0.95(分 tile_face / tile_back 两类)
- **按目标尺寸分桶的召回率与精确率**:<20px / 20~40px / >40px 三桶(较短边),这是本项目最关键的指标
- 不同置信度阈值下的 P-R 数据点(供选线上阈值)
- **按图像来源(source = 博主)分组分别输出全部指标**——读 COCO image 的 `source` 字段;某个博主上指标异常低,说明该域(机位/桌布/牌款)没学好,是补数据的直接信号
- 漏检图清单:把漏检最多的 20 张图可视化(GT 绿框、预测红框、漏检高亮),输出到 `eval_output/missed/`

### 6b. `eval/eval_classification.py` — 分类评测

输入:按类分文件夹的 crop 图 + 模型预测(或直接接 ONNX 模型推理)
输出:
- top-1 准确率(总体 + 每类)
- 29×29 混淆矩阵(热力图 PNG + CSV),**单独列出错误率最高的 10 个混淆对**
- 每类的低置信正确样本和高置信错误样本各存 10 张图,供分析

### 6c. `eval/eval_e2e.py` — 端到端评测

输入:测试集图片 + 检测模型 + 分类模型(支持 Ultralytics .pt 和 ONNX 两种格式,用 `--det-model` / `--cls-model` 传入)
流程:检测 → crop → 分类 → 与 GT 匹配(IoU ≥ 0.5 且类别一致才算对)
输出:端到端准确率(总体 + 按 source 分组)、错误归因分解(漏检 / 误检 / 检对但分类错,三者各占多少)

### 6d. HTML 汇总报告

三个评测脚本的结果汇总渲染成单文件 HTML(`eval_output/report.html`):指标表格 + 混淆矩阵图 + badcase 缩略图墙。用 Jinja2 模板,无需外网 CDN 资源。

**验收标准**:用随机权重模型也能完整跑通三个脚本并出报告(数字难看没关系,管线要通);所有指标计算逻辑有单元测试(构造 3~5 个手工可验算的小例子)。

## 任务 8:基线评测执行

**做什么**:用我现有的 YOLO11 旧模型(我会提供 .pt 文件路径和它的类别名列表)在冻结测试集上跑任务 6 的全套评测,产出第一份 `report.html` 作为项目基线。
注意:旧模型是多类直接检测,需要写一个适配层把它的输出映射到新的两阶段评测协议:万条筒各类映射为 tile_face + 对应分类标签;**若旧模型类别表里有字牌类,其预测一律映射为 tile_face + unknown**。
