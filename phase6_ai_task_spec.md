# Phase 6 需求规格:麻将识别项目 — 实战迭代闭环

> 本文档交给 AI 编码助手(opencode / codex / Claude Code)执行。
> 请严格按任务顺序实现,每个任务有明确的输入、输出和验收标准。
> 本 Phase 依赖:Phase 0~5 全部产物,尤其是 Phase 5 App 的诊断录制能力、Phase 2/3 的 finetune 预设、Phase 0/4 的评测体系。
> 本 Phase 是常态化运转的闭环,不是一次性交付:目标是把"实战发现问题 → 数据回流 → 重训 → 回归 → 发版"压缩成**周级、大部分自动化**的流程。

---

## 项目背景(供 AI 理解上下文)

系统已上线真机(A 轨:眼镜采集 + 手机推理)。实战环境与训练数据(博主视频域)存在残余差距:我自己的眼镜成像、我常去的牌桌/牌款/光照,都是模型没见过的新域。

**最终验收目标(闭环运转 3~4 轮后)**:
- 实战错误率:<1 次/局(错误 = 已确认牌类别错 / 该确认未确认且影响使用)
- 眼镜域冻结测试集(test_set_v2_glasses,任务 2 建立)上,端到端指标达到 Phase 0 博主域测试集同等水平
- 收敛判据:连续 2 轮迭代,v2 测试集提升 <0.3% 且实战错误率不再下降 → 闭环转入低频维护模式(月级)

**闭环全景**:实战佩戴 → 遇错一键诊断录制 → 导回 PC 自动分诊 → 按归因路由到标注/合成补数 → 一键增量重训 → 全量回归门禁 → 发版上机 → 下一轮实战。

### 数据与隐私纪律

- 全部录制数据仅存本地,不上传;录制涉及牌友,开录前需告知并取得同意(写入 runbook 提醒)
- 眼镜域新数据同样遵守泄漏纪律:进入 test_set_v2 的局,整局排除出一切训练用途;实现 `get_leaked_sessions()`(以"局"为隔离单位,粒度比 Phase 1 的视频级更粗更安全)

## 任务 1:闭环脚手架 + 两份操作规程

**做什么**:建目录,写 `field_sop.md` 与 `weekly_playbook.md`。

**`field_sop.md`(实战操作规程)**:
- 开局:征得牌友同意 → App 开始新 session(记录局号)→ 正常游戏
- 遇错即按:发现识别错误,按诊断键(App 已有,录前后 30 秒包),**当下不用做任何分类判断**,继续打牌
- 收局:App 结束 session,记一个主观数字"本局发现错误次数"(哪怕没按键的也估计在内),这是实战错误率的原始数据
- 回家:adb 一条命令导出当日全部 sessions(命令写明)

**`weekly_playbook.md`(每周迭代手册)**:一页纸流程:周中攒数据 → 周末:intake → triage → 读分诊报告(我 10 分钟)→ route 产出标注包(我 0.5~2 小时标注/分拣)→ retrain(挂机)→ regression 看门禁 → 绿灯则 release 上机;每步一条命令,附预计耗时。

**验收**:两份文档可照做;目录完整。

## 任务 2:眼镜域冻结测试集 `test_set_v2_glasses`(本 Phase 第一优先级)

**做什么**:兑现 Phase 0 预留:用最初 2~3 周的实战录制,建立第二个冻结测试集,此后双测试集并行评测。

**功能需求**:
1. 从早期 sessions 中选帧:复用 Phase 0 的抽帧/质量过滤/phash 去重/配额逻辑(经 paths.yaml 调用,不重写),配额维度改为"牌桌场景 × 光照 × 距离",目标 250~350 张
2. 生成 CVAT 任务包(带当前最优模型的预标注,我校正而非从零标)+ 沿用 Phase 0 标注规范
3. 入库:validate_coco 校验 → 冻结目录 + MANIFEST;贡献过帧的局全部进 `get_leaked_sessions()` 名单
4. 接入评测:Phase 0/2/3 的 run_eval 增加 `--dataset frozen_v2` 选项(改动提交回各仓库),同样的确认提示与留痕纪律;HTML 报告支持 v1/v2 并排
5. 立即跑一次当前线上模型在 v2 上的基线评测——v1 与 v2 的指标差距就是"博主域→眼镜域"的 domain gap 量化值,是本 Phase 要消灭的主要目标

**验收标准**:v2 冻结集自包含且零泄漏(校验报告);双测试集并排报告可出;v2 基线数字落档。

**人工配合**:校正 250~350 张预标注,约 3~4 小时(一次性投入)。

## 任务 3:诊断包导入与登记 `scripts/intake.py`

**做什么**:把手机导出的诊断包规范化入库。

**功能需求**:
1. 扫描导出目录,按 session 归档到 `sessions/s{日期}_{序号}/`;校验诊断包完整性(帧+中间结果+事件流,格式同 Phase 4/5 录制规范),损坏包隔离并报告
2. 每个诊断事件(一次按键)在 `registry.jsonl` 追加一条登记:id、session、时间戳、包路径、状态(`new`)
3. phash 级重复检测:同一局对同一问题连按多次的包自动归并为一条(保留全部包引用)
4. 录入我收局时报的"主观错误次数",与按键数一起进台账(两者差值反映漏报率)
5. 幂等:重复运行不重复登记

**验收标准**:导入 3 个真实 session 后台账正确;损坏包被隔离;重复导入无副作用。

## 任务 4:自动分诊 `scripts/triage.py`

**做什么**:对每个 `new` 状态的诊断事件,用 replay 自动复现并归因,产出"我只需读结论"的分诊报告。

**功能需求**:
1. 调 Phase 4 replay 加载诊断包,定位按键时刻前后的事件流异常窗口
2. **归因决策树**(规则进 triage_rules.yaml,可迭代):
   - 检测层:目标处无任何检出(漏检)/ 无牌处有检出(误检)→ 细分尺寸桶、模糊度、遮挡
   - 分类层:检出框正确但投票类别错 → 记录混淆对、crop 尺寸
   - 时序层:单帧多数正确但确认结果错/闪烁/确认超时 → 跟踪断裂(GMC失效?)或投票参数问题
   - 区域层:zone 归类错
   - 不可自动归因 → `manual_review`
3. 每事件产出:归因标签(可多个)+ 关键帧可视化(GT 无,故展示"模型行为时间线":检出/分类/投票/状态逐帧条带图)+ 写回台账(状态 `triaged`)
4. 周报汇总:本周各归因桶计数、与历史周对比趋势、Top 问题模式(如"暗光+牌河远牌漏检占 60%")
5. `manual_review` 队列生成 replay 快捷脚本,我逐个回放人工判因后在台账标注

**验收标准**:对 10 个已知原因的构造案例(从评测 badcase 制作),自动归因准确 ≥8 个;时间线条带图可读;周报趋势正确。

**人工配合**:读周报 10 分钟;manual_review 通常 <5 个/周,每个 2 分钟。

## 任务 5:数据路由 `scripts/route.py`

**做什么**:把分诊结果转化为可执行的补数动作,自动打包,我只做最后的标注/分拣。

**功能需求**:
1. 路由规则(规则化 + 报告中说明理由):
   - 检测漏检/误检类 → 从相关 session 的录制中扩采相似帧(同场景同尺寸桶,phash 多样性),当前模型预标注 → CVAT 增量包 → 标完入 `real_det_v{N+1}`(调 Phase 2 merge_real_data)
   - 分类错误类 → 扩采相关 crop → 预分类分拣包 → 我分拣 → 入 `real_cls_v{N+1}`(调 Phase 3 流程)
   - 时序/参数类 → 不产数据,生成"扫参建议任务"(指向 Phase 4 tune_params,附该 badcase 录制作为扫参数据)
   - 域整体性问题(某牌桌/光照全面差)→ 生成"合成补数建议"(指向 Phase 1:从该域帧提取背景/模板补充素材库,列出具体素材采集脚本命令)
2. 扩采约束:全部经 `get_leaked_sessions()` 过滤;每轮增量标注总量控制(默认 ≤300 张框选 + ≤3000 张分拣,保证我周末 2 小时内完成)
3. 路由完成更新台账状态(`routed`,关联到数据版本号),形成"badcase → 数据 → 模型版本"全链路追溯

**验收标准**:对一份分诊结果运行,产出的 CVAT 包/分拣包可直接使用;台账链路完整;扩采零泄漏。

## 任务 6:一键重训与回归门禁 `scripts/retrain.py` + `regression.py`

**做什么**:把"新数据 → 新模型 → 敢不敢上机"全自动化。

**retrain.py**:
1. 编排:检测 finetune(Phase 2 preset,从当前线上权重起)→ 分类 finetune(Phase 3)→ 量化(Phase 5 quant,沿用既有校准集 + 自动补充新域校准样本)→ 各后端转换 → 全程一条命令挂机,失败可断点续跑
2. 增量原则:默认只 finetune 不全量重训(快,周级可行);每 4~6 轮或 v2 指标停滞时,提示建议全量重训一次(防增量漂移)

**regression.py(发版门禁,红线写死)**:
1. 自动跑全量回归矩阵:frozen v1(不得倒退 >0.3%)+ frozen v2(必须提升或持平)+ Phase 4 视频评测集四项指标(全 PASS)+ 量化对齐(Phase 5 标准)+ 端侧 bench(fps/延迟不倒退,经 adb 自动执行,真机我连好即可)
2. 输出单页门禁报告:每项 PASS/FAIL + 与线上版本对比列;任一 FAIL 则禁止 release 并指出最可疑变更
3. 所有 frozen 评测自动留痕(沿用各 Phase 的 log 纪律)

**验收标准**:全流程(重训→量化→回归)一条命令完成,周末挂机一晚出结果;门禁对"故意做坏的模型"(如旧版权重)能正确 FAIL。

---

## 任务 7:发版回滚与迭代看板 `scripts/release.py` + `dashboard.py`

**做什么**:版本管理收口 + 让迭代是否收敛一目了然。

**release.py**:
1. 门禁全绿后打包 `release_v{N}`(沿用 Phase 5 发布目录规范),自动生成 CHANGELOG(本轮:消化了哪些 badcase id、新增数据版本、指标变化)
2. adb 一键部署到手机;保留上一版本,`--rollback` 一条命令回滚
3. 台账中本轮消化的 badcase 标记 `resolved(v{N})`——下轮实战若同模式问题再现,分诊报告自动标"回归复发"高亮

**dashboard.py**:
1. 生成单页 HTML 看板:按迭代轮次的曲线——实战错误率(次/局,来自我收局上报)、v1/v2 测试集核心指标、各归因桶 badcase 数量趋势、累计数据量
2. 收敛判定:按本文档开头判据自动给出"继续迭代 / 转维护模式"建议
3. 全部数据来自台账与各 Phase 评测留痕日志,无人工填数

**验收标准**:发版/回滚真机演练通过;看板在 3 轮模拟数据下曲线与建议正确。

## 全局技术约束

- Python 3.10+;不引入数据库与 Web 服务,台账用追加式 jsonl + 文件目录(与全项目一致);看板与报告均为静态 HTML
- 一切复用:本 Phase 几乎不写新算法,核心价值是**编排** Phase 0~5 的既有能力,凡能调用既有脚本的绝不重写;对各 Phase 仓库的必要改动(如 frozen_v2 接入)以最小补丁提交回原仓库
- 台账状态机(new→triaged→routed→resolved)与全链路追溯有单测;泄漏过滤(get_leaked_sessions)有单测
- 每轮迭代的全部产物可凭版本号+台账完整复现

## 执行顺序与依赖

任务 1 →(我实战 2~3 周攒首批 sessions)→ 任务 2(v2 冻结集 + domain gap 基线)→ 任务 3 → 任务 4 → 任务 5 → 任务 6 → 任务 7 → 进入周级循环:实战 → intake/triage(自动)→ route + 我标注 → retrain/regression(挂机)→ release → 实战……直至收敛转维护

## 我(人类)负责的部分,你不要尝试做

- 实战佩戴与一键上报;收局上报主观错误次数;取得牌友的录制同意
- v2 测试集校正标注(一次性 3~4 小时);每周增量标注/分拣(≤2 小时/轮)
- 读分诊周报与门禁报告,做发版/回滚/转维护的决策
- manual_review 队列的人工判因
- 最终验收:实战错误率 <1 次/局且连续 2 轮收敛,宣布项目进入维护模式
