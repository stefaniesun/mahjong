## 项目背景(供 AI 理解上下文)

四川麻将识别系统。单帧模型已就绪(检测 2 类 + 分类 29 类),本 Phase 把它们组装成**视频流系统**:跟踪 + 多帧投票 + 状态机,把单帧 ~97% 的不稳定输出变成视频级 ≥99.5% 的稳定输出。

**最终验收目标(在任务 2 构建的视频评测集上)**:
- 稳定牌识别准确率 ≥99.5%(处于"已确认"状态的牌,类别正确率)
- 新牌从出现到"已确认"的延迟 ≤0.5s(15 帧 @30fps 内)
- 零闪烁:已确认牌的类别在其生命周期内变更次数,平均 <0.05 次/牌
- 吞吐:1080p 输入,PC(单消费级 GPU)上端到端 ≥30fps;纯 CPU 模式 ≥10fps(为 Phase 5 端侧可行性探底)

**核心挑战(设计必须针对性处理)**:第一人称视角下**牌基本静止、相机持续运动**——头部转动会让全部框整体平移,普通 ByteTrack 会大量断 track。必须做全局运动补偿(GMC)。

## 任务 1:架构设计文档 + 流水线骨架

**做什么**:先写 `docs/architecture.md` 与 `pipeline.yaml` 骨架,再搭包结构(各模块先放接口与空实现)。

**architecture.md 必须说清**:
1. 数据流图:capture → [每 N 帧] detector → tracker(GMC)→ [对新 track 与未确认 track] classifier → voter → state_machine → events;旁路:可视化与录制
2. 线程模型:capture / GPU 推理 / 后处理三线程 + 无锁队列,队列满丢旧帧(实时优先);确定性回放模式则单线程同步(调试用)
3. **算力分配策略**:检测每 N 帧跑一次(默认 N=3),中间帧用跟踪外推;分类只对"未确认或低置信 track"跑,已确认 track 每 30 帧抽检一次(防换牌场景);这套调度是端侧可行性的关键,参数全部进 pipeline.yaml
4. 各模块接口契约(输入输出 dataclass 定义),保证模块可单独测试、Phase 5 可逐模块替换为 C++/NPU 实现

**验收**:文档评审通过(我读);包骨架 import 无错;接口有类型注解。

## 任务 2:视频评测集构建 `scripts/build_video_testset.py` + 标注指引

**做什么**:建立视频级评测的 GT。逐帧标注成本不可接受,采用**稀疏检查点标注法**。

**功能需求**:
1. 从博主视频中选 8~12 段、每段 30~90 秒的评测片段(必须来自 `get_leaked_videos()` 名单内的视频——即测试集同源视频,它们本来就被训练侧排除,正好复用作视频评测,不浪费;脚本自动校验此约束并防止反向泄漏:这些片段同样不得进入任何训练)
2. 片段要覆盖:静态看手牌 / 出牌动态 / 快速转头 / 远距牌河 / 遮挡频繁,选段时辅助显示旧模型检出密度帮我挑
3. **检查点标注法**:每段每 5 秒冻结一帧作为检查点,我在 X-AnyLabeling 里只标这些检查点帧(完整框+29类,约 6~18 帧/段,共 ~100 帧;脚本先把检查点帧抽成图片目录并生成预标注);另外对每段人工记录 3~5 个"新牌出现事件"(牌、出现的视频时间戳精确到 0.1s),写入 events_gt.json,用于测确认延迟
4. 生成检查点标注目录(图片 + X-AnyLabeling 预标注)与 `checkpoint_annotation.md` 操作指引(含事件时间戳的记录方法:用任意播放器如 PotPlayer 逐帧定位读帧号换算,脚本提供帧号→时间戳换算工具)
5. 导入校验:检查点 GT 通过 Phase 0 validate_coco.py;events_gt.json schema 校验

**验收标准**:评测集构建完成后,目录自包含(片段视频 + 检查点 COCO + 事件 GT + MANIFEST);与训练数据零交集的校验报告。

**人工配合**:标 ~100 个检查点帧 + 记 ~40 个事件时间戳,约 4~5 小时。

## 任务 3:跟踪器 `mahjong_rt/tracker.py`(本 Phase 技术难点)

**做什么**:ByteTrack 改造,适配"目标静止、相机运动、目标数量多且密集"的场景。

**功能需求**:
1. 集成 ByteTrack 核心(两级关联),纯 Python/numpy 实现,不依赖重型框架
2. **全局运动补偿(GMC)**:每帧用稀疏光流(cv2.goodFeaturesToTrack + calcOpticalFlowPyrLK)或 ORB 特征估计帧间单应矩阵,先把所有 track 的预测框按全局运动变换,再做 IoU 关联;特征点要避开检出框区域(牌区域特征会干扰全局估计);估计失败(特征不足/剧烈模糊)时降级为扩大关联门限
3. 卡尔曼参数针对静止目标调整:过程噪声调低,观测主导;track 丢失保留 buffer 默认 30 帧(转头回来还能接上)
4. 密集场景关联保护:相邻牌框 IoU 高,关联加入外观约束开关(用分类器倒数第二层 embedding 的余弦距离,默认关,转头剧烈时自动启用)
5. 输出:每帧的 track 列表(track_id / bbox / det_conf / 状态),ID switch 尽量少

**验收标准**:在"快速转头"评测片段上,与无 GMC 的基线对比,track 断裂数下降 ≥70%(脚本输出对比数据);静止段落 ID switch ≈ 0;单帧跟踪耗时 <5ms(40 个目标,CPU)。

## 任务 4:投票器 `mahjong_rt/voter.py`

**做什么**:把每个 track 累积的多帧分类结果融合成一个稳定类别判定。

**功能需求**:
1. 每 track 维护滑动窗口(默认 7 次分类观测);**加权投票**,权重 = 校准后置信度 × 尺寸权重(crop 越大权重越高,读 calibration.json 的分桶阈值,低于弃权阈值的观测直接丢弃不入窗)
2. 判定规则:窗口内最高票类占比 ≥60% 且有效观测 ≥3 次 → 输出该类与聚合置信度;否则输出"未定"
3. **滞回(hysteresis)**:已有判定的 track,新类别要连续 ≥4 次有效观测压倒旧类别才允许改判(防单帧抖动翻转,同时保留真实换牌场景的纠正能力)
4. back 与牌面互转的特殊处理:牌翻面是真实高频事件(摸牌亮牌),back→具体牌类的改判放宽到 2 次观测
5. 全部参数进 pipeline.yaml;投票逻辑为纯函数式实现(输入观测序列→输出判定),方便单测与离线扫参

**验收标准**:单测覆盖:正常收敛 / 弃权过多输出未定 / 滞回防抖 / 翻面快速改判 四类场景;给定观测序列输出确定可复现。

## 任务 5:状态机与事件流 `mahjong_rt/state_machine.py` + `events.py`

**做什么**:牌的生命周期管理与对外输出协议。

**功能需求**:
1. 状态:`TENTATIVE`(新 track,未满投票条件)→ `CONFIRMED`(投票判定产生)→ `OCCLUDED`(track 暂失,保留)→ `LOST`(超时移除);状态迁移条件全部参数化
2. 对外事件协议(events.py 定义,JSON 序列化):`tile_confirmed`(track_id/类别/置信度/bbox/zone/时间戳)、`tile_updated`(改判)、`tile_lost`、`frame_summary`(每帧全量快照,可关)——**此协议是 Phase 5 端侧与上层应用(胡牌提示等)的接口,字段一旦定稿不轻易改,文档必须完整**
3. 防闪烁兜底:`CONFIRMED` 状态的对外类别由状态机持有,仅在投票器满足滞回条件时更新——可视化与事件流永远不直接暴露单帧结果

**验收标准**:协议文档完整(每字段类型/语义/示例);状态机全迁移路径有单测;录制回放中事件流与画面行为一致(我人工抽查)。

## 任务 6:区域划分 `mahjong_rt/zones.py`

**做什么**:把每张确认牌归入业务区域:`my_hand` / `river`(牌河)/ `opponent_wall` / `meld_area` / `unknown_zone`。

**功能需求**:
1. 启发式 v1(不训模型):综合 (a) 画面位置先验(第一人称下手牌在画面下方 1/4 且框最大)(b) 框尺寸聚类(同区域牌尺寸相近)(c) 排布几何(手牌共线紧密横排,牌河网格,牌墙长条双层)——用 RANSAC 拟合直线/网格归组
2. 输出每张牌的 zone 标签进事件流;不确定给 `unknown_zone`,不强行猜
3. 区域结果做时序平滑(zone 由 track 多数帧投票,不逐帧跳)
4. 在 pipeline.yaml 留 `zones.enabled` 开关——此模块是增值功能,出问题可整体关闭不影响主链路

**验收标准**:在评测片段可视化上,my_hand 与 river 的归类人眼判断准确率 ≥90%;关闭开关后主链路无任何行为变化。

## 任务 7:流水线编排与运行入口 `mahjong_rt/pipeline.py` + `scripts/run_pipeline.py`

**做什么**:把所有模块组装为可运行系统。

**功能需求**:
1. 三线程流水线(按 architecture.md),实时模式丢帧策略 + 确定性回放模式(`--deterministic`,处理每一帧,用于评测与调试,同输入必同输出)
2. run_pipeline.py 模式:`--source 文件/摄像头索引` / `--headless`(无窗口,只出事件流到 stdout 或 jsonl 文件)/ `--record`(录制原始帧 + 每帧全量中间结果到 recordings/,供 replay.py 离线复现任意时刻)
3. 可视化叠加:框 + 确认类别(中文牌名)+ track_id + zone 着色 + 状态色边框(TENTATIVE 黄/CONFIRMED 绿)+ 左上角性能面板(各阶段耗时/fps/丢帧率)
4. 性能埋点:各阶段耗时直方图随 `--record` 落盘
5. `scripts/replay.py`:加载录制,可逐帧步进、跳转、单独重跑某模块(改参后只重算投票与状态机,不重推模型——扫参提速的关键)

**验收标准**:对评测片段实时模式 ≥30fps(GPU)、≥10fps(`--cpu`);确定性模式两次运行事件流逐字节一致;录制回放可复现实时运行的画面与事件。

## 任务 8:视频级评测 `scripts/eval_video.py`(本 Phase 验收标尺)

**做什么**:在任务 2 评测集上自动计算全部验收指标。

**功能需求**:
1. 用确定性模式跑全部评测片段,采集事件流与逐帧状态
2. **检查点准确率**:在每个检查点时刻,取系统当前全部 CONFIRMED 牌与该帧 GT 匹配(IoU≥0.5 + 类别),输出 精确率/召回率/类别准确率,并分尺寸桶
3. **确认延迟**:对 events_gt.json 的每个出现事件,测系统 `tile_confirmed` 时间戳差,输出 p50/p95
4. **闪烁率**:统计每个 CONFIRMED track 生命周期内 `tile_updated` 次数,输出平均与最差样例(可视化定位)
5. **跟踪质量**:借检查点 GT 估算 ID switch 与 track 断裂
6. HTML 报告(沿用 Phase 0 报告风格)+ 与上一次评测的对比列
7. 验收红线检查:对照本文档开头四项指标,逐项 PASS/FAIL 醒目输出

**验收标准**:指标计算逻辑全部有单测(构造已知事件序列验算);报告可读;两次评测(确定性)数字完全一致。

## 任务 9:离线参数扫描 `scripts/tune_params.py`

**做什么**:对投票窗口、占比阈值、滞回次数、检测间隔 N、track buffer 等 8~10 个关键参数做网格/随机搜索,在录制数据上离线寻优。

**功能需求**:
1. 基于 replay 的"只重算后处理"能力,单组参数评测 <30 秒;支持并行
2. 目标函数与约束:最大化检查点准确率,约束确认延迟 p95 ≤0.5s、闪烁率达标;输出 Pareto 前沿表与推荐参数组
3. 推荐参数写回 pipeline.yaml 的候选区(不自动覆盖现行配置,我确认后手动启用)
4. 防过拟合:扫参只允许用片段的一半(脚本强制划分 tune/holdout),最终用 holdout 半区验证推荐参数

**验收标准**:一轮 200 组参数扫描在 2 小时内完成;tune/holdout 划分强制生效;推荐参数在 holdout 上指标不劣于默认参数。

## 全局技术约束

- Python 3.10+;核心依赖:opencv-python, numpy, onnxruntime(-gpu), pyyaml, tqdm, jinja2;**不引入 torch**(本 Phase 只消费 ONNX,保持与 Phase 5 端侧依赖形态一致)
- ByteTrack 自含实现(参考开源,纯 numpy),不引入整套 MOT 框架
- 核心算法模块(tracker/voter/state_machine/zones)零 GPU 依赖、纯函数优先、全部可单测;pipeline 层才出现线程与 IO
- 所有随机性可被 seed 固定;确定性模式是硬要求,贯穿所有模块设计
- 实验/录制目录自包含规范同 Phase 2/3
